# 测试驱动开发

## 理念

**核心原则**：测试应通过公开接口验证行为，而非实现细节。代码可以彻底重写，测试不应受影响。

**好的测试**是集成风格的：它们通过公开 API 执行真实的代码路径。它们描述系统**做什么**，而不是**怎么做**。好的测试读起来像规格说明——"用户可以用有效购物车结账"准确告诉你系统具备什么能力。这些测试能在重构中存活，因为它们不关心内部结构。

**坏的测试**与实现耦合。它们 mock 内部协作者、测试私有方法，或通过外部手段验证（比如直接查询数据库而不是使用接口）。警告信号：重构时测试挂了，但行为没变。如果你重命名一个内部函数导致测试失败，那些测试测的是实现，不是行为。

参见 [tests.md](tests.md) 了解示例，[mocking.md](mocking.md) 了解 mock 指南。

## 反模式：水平切片

**不要先写完所有测试，再写所有实现。**这就是"水平切片"——把红灯阶段当成"写所有测试"，把绿灯阶段当成"写所有代码"。

这会产生**糟糕的测试**：

- 批量编写的测试测的是_想象中的_行为，而非_实际_行为
- 你最终测的是事物的_形状_（数据结构、函数签名），而非面向用户的行为
- 测试对真实变化不敏感——行为坏了测试照过，行为没问题测试反而挂了
- 你跑在灯光前面了，在理解实现之前就锁定了测试结构

**正确做法**：通过曳光弹进行垂直切片。一个测试 → 一个实现 → 重复。每个测试都基于上一轮循环中学到的东西。因为你刚写完代码，你确切知道什么行为重要、如何验证。

```
错误（水平）：
  红灯：test1, test2, test3, test4, test5
  绿灯：impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  红灯→绿灯：test1→impl1
  红灯→绿灯：test2→impl2
  红灯→绿灯：test3→impl3
  ...
```

## 工作流

### 1. 规划

写任何代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认测试哪些行为（排优先级）
- [ ] 识别[深模块](deep-modules.md)的机会（小接口，深实现）
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

提问："公开接口应该是什么样的？哪些行为最重要需要测试？"

**你不可能测试所有东西。**与用户确认哪些行为最重要。把测试精力集中在关键路径和复杂逻辑上，而不是每个可能的边界情况。

### 2. 曳光弹

写一个测试，确认系统的一件事：

```
红灯：为第一个行为写测试 → 测试失败
绿灯：写最少的代码让测试通过 → 测试通过
```

这是你的曳光弹——证明路径端到端可行。

### 3. 增量循环

对每个剩余行为：

```
红灯：写下一个测试 → 失败
绿灯：写最少的代码让测试通过 → 通过
```

规则：

- 一次一个测试
- 只写刚好让当前测试通过的代码
- 不要预判未来的测试
- 保持测试聚焦于可观察的行为

### 4. 重构

所有测试通过后，寻找[重构候选](refactoring.md)：

- [ ] 提取重复代码
- [ ] 加深模块（把复杂性隐藏在简单接口后面）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 思考新代码揭示了现有代码的什么问题
- [ ] 每次重构步骤后运行测试

**红灯时绝不重构。**先到绿灯。

## 每轮检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只使用公开接口
[ ] 测试能在内部重构中存活
[ ] 代码对当前测试来说是最少的
[ ] 没有添加投机性功能
```
